# Next.js Linear-style landing page

A greenfield Next.js 15 landing page mimicking Linear.app's dark aesthetic: fixed glassmorphism navbar with mobile hamburger toggle, gradient-text hero with animated glow orbs, a six-card features grid with hover states, and a multi-column footer. Built on Tailwind CSS v4 (using the new `@theme` directive and `@tailwindcss/postcss` plugin), TypeScript, and the App Router. All pages compile as static output.

**Watch for:** Mobile menu doesn't close on link tap (confirmed), nav links duplicated between desktop and mobile markup (confirmed), no keyboard/focus management on the mobile menu (confirmed), and zero test coverage for the only interactive component.

## High-level view

Only the Navbar carries client-side state (a boolean for mobile menu visibility); everything else is server-rendered markup with CSS-only interactivity. The Tailwind v4 integration uses `@theme` inside `globals.css` to define custom keyframes and color tokens—no separate `tailwind.config.ts` exists, which is the intended v4 pattern.

The Navbar toggles a mobile drawer via `useState`. The toggle has an accessibility gap (no focus trap, no `aria-expanded`, no Escape-to-close) and a UX gap (tapping a link doesn't dismiss the menu). These are the most actionable issues in the change.

<details>
<summary>Issues (5)</summary>

1. **Mobile menu stays open on navigation** — Tapping an anchor link (e.g. `#features`) scrolls the page but the menu remains open. Add an `onClick` handler on each mobile link that calls `setMobileOpen(false)`.
2. **Missing `aria-expanded` on hamburger toggle** — The mobile menu button has `aria-label` but no `aria-expanded={mobileOpen}`, so screen readers can't determine menu state.
3. **No Escape-key dismissal for mobile menu** — Standard mobile nav UX closes on Escape. Add a `useEffect` keydown listener when `mobileOpen` is true.
4. **Nav link duplication** — Desktop and mobile nav links are hardcoded separately. A shared array (like the pattern used in `Features.tsx`) would prevent link drift between the two.
5. **No tests for Navbar interactivity** — The only stateful component has no unit or integration tests. A simple React Testing Library test confirming toggle visibility and link-close behavior would catch regressions.

## Mobile menu lifecycle

When a user taps a link inside the mobile menu (e.g. the `#features` anchor), the browser scrolls to the target but `mobileOpen` remains `true`. The drawer stays visible over the scrolled content. The fix is trivial—call `setMobileOpen(false)` in an `onClick` on each mobile link—but its absence means the mobile experience is broken on any page with in-page anchors.

There's also no keyboard escape hatch. A focused user who opens the menu via keyboard has no way to dismiss it except re-activating the hamburger button. Adding an effect that listens for `keydown` on `Escape` when the menu is open is standard practice.

## Accessibility posture

The hamburger button is missing `aria-expanded={mobileOpen}` and `aria-controls` pointing to the menu panel's `id`. Without `aria-expanded`, assistive technology can't communicate the menu's current state. The mobile drawer itself has no role or focus management—when it opens, focus doesn't move into it, and there's no trap preventing focus from wandering behind the overlay.

The feature cards use emoji for icons via raw Unicode characters in JSX. Some screen readers will announce these literally ("lightning bolt", "sparkles", etc.), which is noisy. Wrapping them in a `<span aria-hidden="true">` and relying on the following `<h3>` for the card's accessible name would clean this up.

## Duplicated link definitions

The Navbar declares its navigation links twice: once in the desktop section and once in the mobile drawer. The two lists are raw JSX without a shared data source. Compare this to `Features.tsx`, which maps over a `features` array—a pattern that prevents silent drift. Extracting `const navLinks = [...]` and mapping in both locations would prevent the lists from diverging.

## Tailwind v4 theme — dead code

The color tokens (`--color-purple-400` through `--color-indigo-600`) in the `@theme` block re-declare values that match Tailwind v4's default palette—they're effectively no-ops that add noise without extending anything.

`animate-slide-up` is defined in the theme but never referenced in any component—dead CSS that should be pruned or used.

## Missing `prefers-reduced-motion`

The two fixed-position gradient orbs in `page.tsx` use `blur-3xl` with continuous `animate-pulse-glow`. On low-end mobile devices, large blurred elements compositing on every frame can cause jank. A `prefers-reduced-motion` media query disabling the animation would be a progressive enhancement:

```css
@media (prefers-reduced-motion: reduce) {
  .animate-pulse-glow {
    animation: none;
  }
}
```

## Test coverage

There are no tests. The Navbar is not purely static—it has interactive state. A minimal test asserting that the mobile menu becomes visible on button click, and that it closes when a link is tapped, would validate the behavioral gaps flagged above.
